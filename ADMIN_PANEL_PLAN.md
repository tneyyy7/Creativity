# План развития админ-панели

> Документ описывает поэтапное улучшение и расширение админ-панели проекта Creativity.
> Каждый этап содержит цели, конкретные шаги, затрагиваемые таблицы/файлы и подводные камни.

## Текущее состояние (как есть)

- **Один экран** — очередь жалоб: `src/pages/Admin.jsx` (~242 строки).
- **API модерации** — `src/lib/api/moderation.js`.
- **Гейтинг доступа** — `fetchIsAdmin(userId)` читает `profiles.is_admin`; пункт меню добавляется в `src/components/Sidebar.jsx` при `isAdmin === true`; роутинг во вкладке `admin` в `src/App.jsx`.
- **Существующие действия:**
  - удалить пост — `adminDeletePainting(id)`
  - удалить комментарий — `adminDeleteComment(id)`
  - бан/разбан юзера — `setUserBanned(userId, banned)`
  - смена статуса жалобы — `updateReportStatus(reportId, status)` (`pending` / `reviewed` / `dismissed`)
- **Связанные миграции:** `supabase/migrations/phase1_admin.sql`, `phase1_moderation.sql`.

### Чего НЕТ
- Дашборда / обзорных метрик.
- Управления пользователями вне контекста жалобы.
- Проактивной модерации контента (только реактивная — по жалобам).
- Видимости подписок / биллинга.
- Журнала действий администраторов (audit log).
- Ролей — только булев флаг `is_admin` (всё или ничего).

### Доступные таблицы (модель данных)
`profiles`, `paintings`, `post_comments`, `post_likes`, `bookmarks`, `collections`, `collection_paintings`,
`stories`, `story_likes`, `story_views`, `follows`, `friendships`, `blocked_users`,
`messages`, `group_chats`, `group_members`, `chat_mutes`, `chat_pins`, `chat_hides`, `user_chat_themes`,
`notifications`, `reports`, `subscriptions`, `pro_profile_settings`, `tags`, `tag_follows`, `painting_tags`,
`custom_emojis`, `avatars`.

`create_stripe_checkout`, `stripe_webhook`, `subscription_webhook`, `onesignal-notify`.

## Этап 1. Фундамент (инфраструктура админки)

> Без этого опасно расширять: сейчас любой `is_admin` может всё, и нет следов действий.

### Цели
1. **Layout с вкладками** вместо одного экрана.
   - Превратить `Admin.jsx` в контейнер с табами: `Dashboard / Reports / Users / Content / Subscriptions / Logs`.
   - Вынести текущую очередь в `src/pages/admin/ReportsQueue.jsx`.
   - Каждая вкладка — отдельный lazy-компонент.
2. **Audit log** — таблица `admin_actions`:
   - Поля: `id`, `admin_id`, `action`, `target_type`, `target_id`, `meta jsonb`, `created_at`.
   - Каждое действие (бан, удаление, смена статуса, выдача Pro) пишет запись.
   - Запись лучше делать на стороне БД (триггер или RPC), чтобы её нельзя было обойти из клиента.
3. **Роли вместо булева флага** — `profiles.admin_role` enum:
   - `moderator` — видит и обрабатывает жалобы, модерирует контент.
   - `admin` — + управление пользователями и подписками.
   - `superadmin` — + выдача/снятие ролей, удаление аккаунтов, доступ к логам.
   - `is_admin` оставить для обратной совместимости, постепенно мигрировать.
4. **RLS-аудит** — проверить, что все admin-операции защищены политиками на стороне БД, а не только скрытием кнопки в UI.

### Затрагиваемые файлы
- `src/pages/Admin.jsx` → рефактор в layout.
- `src/pages/admin/*` → новые компоненты вкладок.
- `src/lib/api/moderation.js` → добавить `logAdminAction`, `fetchAdminRole`.
- Новая миграция `supabase/migrations/phase3_admin_foundation.sql`.

### ⚠️ Подводные камни
- **RLS — главный риск.** Скрыть кнопку в UI ≠ запретить операцию. Любой залогиненный юзер может вызвать Supabase напрямую. КАЖДОЕ admin-действие должно проверяться политикой в БД (`auth.uid()` имеет нужную роль).
- **Рекурсия в RLS-политиках.** Политика на `profiles`, которая сама читает `profiles.admin_role`, может уйти в рекурсию. Решение — `SECURITY DEFINER` функция `is_admin(uid)` / `has_role(uid, role)`, помеченная как стабильная, и использовать её в политиках.
- **Совместимость колонок.** В коде уже есть паттерн «колонка может не существовать → graceful degrade» (см. `fetchIsAdmin`, `fetchBlockedIds`). Сохранять этот паттерн: новые колонки/таблицы не должны ронять основную загрузку приложения, если миграция ещё не применена.
- **Логирование не должно блокировать действие.** Если запись в `admin_actions` упала — само действие (бан и т.п.) уже выполнено. Логировать в той же транзакции/RPC, либо не прерывать UX при ошибке лога.
- **Понижение собственной роли.** Superadmin не должен случайно снять роль сам у себя и заблокировать доступ. Защита: нельзя снять последнего superadmin'а.

## Этап 2. Dashboard (обзорный экран)

### Цели
Стартовая вкладка с метриками по существующим таблицам:
- **Карточки-счётчики:** всего юзеров, новые за 7 дней, активные подписки (и MRR из `subscriptions`), постов/историй за сутки, **открытых жалоб** (бейдж).
- **Мини-графики роста:** регистрации (`profiles.created_at`), посты (`paintings.created_at`).
- **«Требует внимания»:** pending-жалобы, недавние баны, недавние отмены подписок.

### Затрагиваемые файлы
- `src/pages/admin/Dashboard.jsx` (новый).
- `src/lib/api/adminStats.js` (новый) — агрегирующие запросы.

### ⚠️ Подводные камни
- **Не агрегировать на клиенте по 10к+ строк.** `select('*')` и подсчёт в JS не масштабируется. Использовать `count: 'exact', head: true` для счётчиков и RPC/SQL-вью для агрегаций по датам.
- **Тяжёлые запросы при каждом заходе.** Метрики кэшировать (минимум — на время сессии вкладки), либо считать материализованным вью / по расписанию.
- **Часовые пояса.** «За сутки / 7 дней» — фиксировать UTC или TZ юзера явно, иначе границы дня плывут.
- **MRR-подсчёт.** Учитывать только активные (не canceled/past_due) подписки; разные тарифы → разные суммы.

## Этап 3. Управление пользователями

### Цели
Новая вкладка `Users` (сейчас юзера можно тронуть только если на него есть жалоба):
- **Поиск/таблица** по `profiles`: nickname, email, дата регистрации, pro-статус, бан, роль.
- **Действия:** бан/разбан, выдать/снять `admin_role`, просмотр профиля, удаление аккаунта.
- **Карточка юзера:** его посты, жалобы на него и от него, статус подписки, последняя активность.
- **Ручная выдача Pro** (саппорт / компенсации).

### Затрагиваемые файлы
- `src/pages/admin/Users.jsx`, `src/pages/admin/UserCard.jsx` (новые).
- `src/lib/api/adminUsers.js` (новый).

### ⚠️ Подводные камни
- **Email лежит в `auth.users`, не в `profiles`.** Доступ к нему — через `auth.admin` API (service role), который **нельзя** дёргать с клиента. Нужна edge-функция с service-role ключом. Никогда не выносить service-role ключ в браузер.
- **Удаление аккаунта — каскад.** Снести юзера = снести его посты, лайки, комменты, сообщения, подписки, истории. Продумать ON DELETE CASCADE либо мягкое удаление (`deleted_at`). Жёсткое удаление необратимо — добавить подтверждение и аудит.
- **Удаление в Supabase Auth.** Удаление строки `profiles` не удаляет запись в `auth.users` — нужна edge-функция (`auth.admin.deleteUser`).
- **Пагинация и поиск.** Таблица юзеров будет расти. Серверная пагинация + поиск по индексу (ILIKE по nickname с trigram-индексом), не грузить всё.
- **Эскалация привилегий.** Только `superadmin` может выдавать роли. Модератор/админ не должны иметь возможность повысить себя — проверять на стороне БД.
- **Отмена Pro вручную рассинхронит Stripe.** Если выдавать/снимать Pro в обход Stripe — webhook потом перезапишет. Чётко разделить «ручной grant» и «Stripe-подписку» (отдельный флаг-источник).

## Этап 4. Управление контентом

### Цели
Вкладка `Content` — проактивная модерация, а не только по жалобам:
- **Лента всего контента** (посты/истории) с фильтром: NSFW, по автору, по дате.
- **Bulk-действия,** переключение NSFW-флага, удаление.
- **Управление тегами/хэштегами** (`tags`): мерж, переименование, бан спам-тегов.

### Затрагиваемые файлы
- `src/pages/admin/Content.jsx`, `src/pages/admin/Tags.jsx` (новые).
- Расширение `src/lib/api/moderation.js` (bulk-delete, toggle NSFW).

### ⚠️ Подводные камни
- **Удаление файлов из Storage.** Удаление строки `paintings` не удаляет картинку из бакета. Нужно чистить Storage, иначе мусор копится и растёт счёт за хранилище.
- **Истории истекают сами.** У `stories` есть TTL/просрочка — не дублировать логику истечения, учитывать уже истёкшие.
- **Мерж тегов — переписывание связей.** Слияние тега A в B = массовый апдейт `painting_tags` + `tag_follows`; делать в транзакции/RPC, иначе частичный мерж.
- **Bulk-операции и лимиты.** Supabase ограничивает размер запроса; большие пакеты бить на чанки.
- **Денормализованные счётчики.** В проекте есть `phase1_denormalized_counters` — при удалении контента счётчики (лайки/посты у профиля) надо корректировать, иначе разъедутся.

## Этап 5. Подписки и биллинг

### Цели
Вкладка `Subscriptions` (Stripe есть, но 0 видимости в админке):
- **Список** активных/отменённых подписок из `subscriptions`.
- **Детали:** статус Stripe, дата окончания, история.
- **Действия:** ссылка в Stripe Dashboard, ручная отмена / запуск рефанда.

### Затрагиваемые файлы
- `src/pages/admin/Subscriptions.jsx` (новый).
- `src/lib/api/adminBilling.js` (новый).
- Возможно — edge-функция для безопасных операций со Stripe.

### ⚠️ Подводные камни
- **Stripe-операции только через сервер.** Секретный ключ Stripe нельзя в браузер. Отмена/рефанд — через edge-функцию (по образцу существующих `stripe_webhook` / `create_stripe_checkout`).
- **Источник истины — Stripe, не БД.** `subscriptions` в Supabase — отражение вебхуков и может отставать. Для критичных действий сверяться со Stripe API.
- **Идемпотентность.** Рефанд/отмена не должны выполниться дважды при двойном клике — Stripe idempotency keys + блокировка кнопки.
- **Webhook перезапишет ручное изменение.** Любое ручное изменение статуса в БД будет перетёрто следующим вебхуком — не менять статус в обход Stripe.

## Этап 6. Качество жизни (улучшение текущей очереди жалоб)

### Цели
- **Bulk-действия** по жалобам (выделить → reviewed / dismiss).
- **Группировка:** несколько жалоб на один таргет — одной карточкой со счётчиком.
- **Уведомления юзеру** при бане/удалении (через `notifications` + `onesignal-notify`).
- **Realtime-бейдж** новых жалоб в сайдбаре (Supabase realtime).
- **Reason-фильтр и поиск** по очереди.

### Затрагиваемые файлы
- `src/pages/admin/ReportsQueue.jsx` (вынесенный из текущего `Admin.jsx`).
- `src/components/Sidebar.jsx` — realtime-бейдж.

### ⚠️ Подводные камни
- **Push-функция деплоится особым образом.** `onesignal-notify` деплоить с `--no-verify-jwt` (webhook не шлёт auth-заголовок). См. память проекта.
- **OneSignal Service Worker.** Push использует SW OneSignal на scope `/`; нельзя регистрировать второй SW там же — это убьёт все пуши. См. память проекта.
- **Группировка не должна терять отдельные жалобы.** Объединяя карточки, сохранять связь со всеми исходными `reports` (чтобы все закрылись при действии).
- **Realtime-подписки чистить.** Подписка на новые жалобы должна отписываться при размонтировании, иначе утечки/дубли соединений.
- **Дублирование уведомлений.** Не слать юзеру 5 уведомлений за 5 сгруппированных жалоб — одно событие = одно уведомление.

## Сквозные правила (для всех этапов)

1. **Безопасность на стороне БД, не в UI.** Скрытие кнопки — косметика. Каждое admin-действие защищается RLS-политикой или edge-функцией с проверкой роли. Клиент нельзя считать доверенным.
2. **Service-role ключи и секреты (Stripe, auth.admin) — только в edge-функциях.** Никогда в браузерном бандле.
3. **Graceful degrade при неприменённых миграциях.** Сохранять существующий паттерн: новая колонка/таблица не должна ронять загрузку приложения (try/catch → дефолт), как в `fetchIsAdmin`.
4. **Всё логировать в `admin_actions`.** Любое разрушающее/привилегированное действие оставляет аудит-след.
5. **Подтверждение для необратимого.** Удаление контента/аккаунта, бан, рефанд — через `confirm` и/или модалку. Уже используется `window.confirm` в `Admin.jsx`.
6. **Чистить связанные ресурсы.** Удаление сущности → чистка Storage-файлов, корректировка денормализованных счётчиков, каскад связей.
7. **Серверная агрегация и пагинация.** Не тянуть большие выборки на клиент; `count`-запросы, RPC, индексы, пагинация.
8. **i18n.** Все строки через `t()` (ru/enUS) — проект двуязычный, см. `src/i18n/config.js`.
9. **Стиль/UI единообразно.** Использовать существующие паттерны: `glass-card`, lucide-иконки, классы Tailwind, `ProfileAvatar`. Не плодить новый дизайн-язык.
10. **Lazy-загрузка вкладок.** Тяжёлые админ-экраны грузить через `lazy()`, как уже сделано для `Admin` в `App.jsx`.

## Рекомендуемый порядок реализации

**Этап 1 → 2 → 3** дают максимум ценности: безопасность (логи/роли) + видимость (дашборд) + контроль над пользователями.
**Этапы 4 → 5 → 6** — итеративно после фундамента.

| Приоритет | Этап | Почему |
|-----------|------|--------|
| 🔴 Высокий | 1. Фундамент | Безопасность и аудит — основа всего остального |
| 🔴 Высокий | 2. Dashboard | Быстрый видимый результат, видимость состояния |
| 🟠 Средний | 3. Пользователи | Реальный контроль над аудиторией |
| 🟡 Ниже | 4. Контент | Проактивная модерация |
| 🟡 Ниже | 5. Подписки | Видимость биллинга |
| 🟢 Полировка | 6. QoL очереди | Улучшение того, что уже работает |
